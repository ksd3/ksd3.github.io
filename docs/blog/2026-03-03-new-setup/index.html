How I’d set things up on a new Linux machine
Published

March 3, 2026

A lot of new tools supporting my workflow have been released over the past few years. The majority of a typical day for me is spent writing Markdown, Python3, and C++ code. Given that my day job used to involve reproducing and modernizing dozens of broken deep learning repositories, it’s no surprise that the combination of dependency management hell, obfuscated (often on purpose) code, and the authors’ general ineptitude in writing documentation resulted in a lot of frustration. A general situation of this kind is usually like this:

  • I have a link to a GitHub repository that claims to be the official implementation of one of three or four papers claiming to be SOTA on a particular problem. This repository has a README.md usually structured as follows:

👨‍🏫Quick Start

These are the instructions we give our interns, who work on our machines, with our software setup, to reproduce our work.

Set up the environment

Use conda and then activate the environment. Also pip install -r requirements.txt

Dependency management

Go to (some third party website) and download (5-10 files that require you to spend a significant amount of time downloading them).

Pretrained models

Download pretrained models from (Google Drive/Baidu/EUDAT/HuggingFace/Zenodo/Git LFS storage). But you can also use our handy script to download them! Just execute download_models.sh that may or may not require elevated user access (not supported on many industry machines) and also invoke access tokens located at ~/.folder/token.json that you need to set up before and may scrape websites you’ve never heard of!

Then move them to ./models.

🎬Inference

If you want to test the inference (which you don’t need to download datasets for), run

scripts/inference/single_inference/test_woYtKDTRVGlwCreI6xWDQjWu_15_28.sh
scripts/inference/single_inference/test_woYtKDTRVGlwCreI6xWDQjWu_15_28_big.sh

Also you won’t be able to run these scripts on your own. You have to replace ${some-abbreviated-variable-denoting-preprocessing-type} and ${some-abbreviated-variable-denoting-file-path} in those scripts to process the input data (often processed with another deep learning model) correctly. You can also not use them, just comment them out! (And you won’t get the results we show in our paper..)

You can also benchmark the models by running these scripts:

scripts/inference/batch_inference/test_woYtKDTRVGlwCreI6xWDQjWu_15_28.sh
scripts/inference/batch_inference/test_woYtKDTRVGlwCreI6xWDQjWu_15_28_big.sh

The input data (that’s processed from your original data) is saved to folder_that_was_just_created/real_processed_data.

🚀Train your own models

Here’s how you can train your own models.

Train the preprocessing models

Our preprocessing models have been trained on multiple GPUs.

scripts/train/train_preprocessor.sh

But you can also train on a single GPU.

scripts/train/train_preprocessor_single.sh

And you have several options while training this preprocessor. You can turn these options off by deleting ${variable1}, ${variable2}, and ${variable3}

Train the main model

You should run this command to extract the processed data obtained from the model you just trained. (Often times, this command will retrain the entire model, making the previous command useless)

scripts/train/train_preprocessor_get_codes.sh

You can then train the small or big models.

scripts/train/train_woYtKDTRVGlwCreI6xWDQjWu_15_28.sh
scripts/train/train_woYtKDTRVGlwCreI6xWDQjWu_15_28_big.sh
Evaluate the models

You can then evaluate the models with

scripts/eval/eval_preprocessor.sh
scripts/eval/eval_woYtKDTRVGlwCreI6xWDQjWu_15_28.sh
scripts/eval/eval_woYtKDTRVGlwCreI6xWDQjWu_15_28_big.sh

🚨Postprocessing

You can postprocess the results of the inference. You can adjust ${postprocessing_variable} to get somewhat better results.

cd postprocess/scripts
scripts/run_postprocessing.sh

There are a myriad of problems with this kind of README.md. For the sake of brevity, here are a few of them:

  • Nowhere - not a single time - does the README mention that you must clone it and then set up the virtual environment. This may sound resoundingly obvious, but I have run into too many repositories filled with nonsensical files that actively hinder you from using it.
  • Nowhere - not a single time - does the README mention how the training was done, any nuances, whether they averaged over several seeds, how they minimized randomness.
  • Nowhere - not a single time - does the README mention what the raw output looks like before postprocessing.
  • Nowhere - not a single time - does the README mention how to visualize outputs in different formats.
  • Nowhere - not a single time - does the README mention that your data goes through several stages of preprocessing before its actually fed into the trained model. While this may be obvious to people in the field, it’s still hard to determine what’s going on with your data.
  • Nowhere - not a single time - does the README mention any parameters used to tweak the pre and postprocessing of data. This information is often times not mentioned in the paper either.
  • Nowhere - not a single time, does the README mention reliance on a ten-year-old package that needs manual source editing to work with modern pipelines.

The kicker? The last update to the repository was eight months ago, and despite many people creating issues showing bad the results are from the authors’ pretrained models, or asking for access to data, the authors never respond.

I could go on a rant about the reproducibility crisis in science, but that’s not the point of this blog. However I do understand why people do it. At the student level, you don’t want to get scooped because your future career depends on it. At the corporate level, model architectures, feature engineering tricks, and training pipelines are often treated as proprietary assets, leading to deliberate obfuscation. Scientists are rewarded for novel results, not clean, documented, reusable code, so there is little incentive to invest in clarity and every incentive to keep a competitive edge. There is also the possibility that many people are just bad at coding, so they write sloppy code. Given the amount of moving parts in a typical deep learning pipeline (GPU behavior, random seeds, data loaders), small, undocumented implementation choices can change results, and opaque code makes it nearly impossible to track these down. Finally, groups often use clusters (which take a long time to update their software versions) to compute, so there’s no point in writing code for modern versions of the software stack.

I like using modern software. I think software that’s clean, well-implemented, open-source, and modernized offers a myriad of advantages over older versions of the code. For example, Python 3.14 removes the GIL, so there will finally be support for true multithreading. This greatly simplifies creating efficient PyTorch dataloaders, but I expect it to take a couple of years before PyTorch natively supports this. Given that I expect to be developing on Linux machines for the foreseeable future, I want to use a setup stack that leverages modern open source tools and configure it to my taste.

There are of course some caveats to this. Open-source software controlled by corporations can be monetized. Take the case of MySQL. Oracle copyrights MySQL but maintains an open-source version for community use. However, the community version of MySQl does not have TDE or thread pooling for hundreds of thousands of edition. If you’re ever in a situation where you require your MySQL database to handle a large amount of transactions, you’re forced to fork over the money for a license. On the other hand, you do (in theory) get the benefits of training programs, technical support, and security fixes.

For personal projects, I prefer my software to be open-source (this is due to years spent fighting MATLAB). However, I understand that without a dedicated team supporting and updating the software the user experience is horrible. I will admit that I, like many others, prefer a solid user experience more than I’d like to admit. Given that I work mostly in Python, I run into most of the usual problems (paraphrasing from this):

  • There are a lot of ways to install Python, depending on the OS; each method comes with its own gotchas.
  • There’s a lot to know upfront about using Python, a language designed for beginners.
  • Python is extremely versatile and is used in so many different contexts that it’s hard to know what’s ‘good’ and what’s ‘bad’.
  • Many people claim to give good advice on the matter.
  • There are many tools to solve Python’s problems, so we suffer from the paradox of choices.
  • -m and py are hardly used.
  • Many popular packages are compiled!
  • PYTHONPATH and PATH and general bad naming conventions make setting up Python hard to use across different machines.

I recommend reading the rest of the article to gain a more nuanced understanding of uv. But I’ve switched to it as my package manager anyway. It’s perfect for someone like me, who writes a lot of greenfield projects. Like the article, however, I will also address the elephant in the room. uv’s owner Astral is a commercial entity with no obvious path to profitability. If they’re aiming to become a direct competitor to Anaconda, they must eventually include a GUI as the vast majority of Python users run Python on Windows. However, uv is so good in every other possible way that I don’t see myself switching off it any time soon.

I also care a lot about code readability and maintainability. I care about well-formatted, heavily-commented, and straightforward code. Astral’s other product, Ruff is a code linter that’s extremely easy to use and extremely fast. It replaces all of the other linters I’ve used (like Black, Flake8, isort - (un)ironically, Ruff advertises itself as alternatives to all of them) and works blazingly fast right out of the box.
